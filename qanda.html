Questions and answers

Should pop-ups be locked or should a user be able to move them?
Pop-ups should be movable.

Should pop-ups include a cancel button as well as the x to close the pop up or just one or the other? If so, which?
All pop-ups should have a Save button and a Cancel button by default, unless otherwise specified in the requirements. The positive action (green) should be first (on the left).

When you have a popup screen as the result of an edit function – should it show all fields with those that can’t be updated locked, or show just those fields that are updatable?
Edit screens should show all relevant information. This may include information which cannot be updated and is locked.

When you have a popup open and don’t close it but click onto the screen behind, should the popup remain open or should it close?
The user should not be able to click on the screen from which the pop up was triggered. The underlying screen should remain disabled while the pop-up is open, and the pop-up should contain an explicit action to close it.

When you click from a main screen into a pop-up via a function, when you leave the popup should the main page go back to the top or remain where it was when you selected the function next to e.g. a site?
It should remain where it was.

Layout of data entry screens and controls used – should we use pop-ups, separate screens, in-grid editing etc?
Pop-ups and separate screens are design decisions. Both have their place. A rule of thumb is to look at the amount of data that needs to be entered by the user. For example, if a large amount of data is to be added a separate screen may be more appropriate than a pop up. In-grid editing should be very carefully considered as this has the tendency to be non-standard web behaviour.

HTML Tables! What about controls?
The use of third-party controls should be carefully considered. For example, the removal of Telerik controls in Greenlight Grower Management and Greenlight Supplier Exchange has provided performance improvements, better control of page styling and also reduced the risk of incompatibility with new browsers.

I’ve just created a beautifully smooth transition on a lightbox, can I work this straight into a design?
We’re always looking for new techniques and design ideas but please flag it up first so that it can be reviewed. It may be that it can be used across all products but we will need to engage all product owners first.

Styling – Is there any scope for a customer logo? I know a number of customers are used to this and it does help test/support differentiate deployments.
Positioning of customer logos is a design decision – and may also be a commercial decision - but the use of a customer logo, a Muddy Boots logo and a Greenlight logo on the same page should be avoided so as not to overcrowd the page with logos.

Accessibility standards – should we always try and use alt tags?
Yes – accessibility standards should be adhered to. Accessibility standards are out of scope for this document but a good place to start is http://www.w3.org and their guidelines on accessibility: http://www.w3.org/WAI
